# Provenance Guard

A content-attribution API for a writing platform. It accepts a piece of text,
estimates whether it was written by a human or generated by AI, and returns a
**confidence score** plus a **plain-language transparency label** that a
non-technical reader can understand. Creators can **appeal** a decision, and
every decision and appeal is written to a structured **audit log**.

The system fuses two independent detection signals rather than trusting a single
model, separates the *direction* of a prediction from the *confidence* in it,
and is deliberately tuned to make "AI" harder to declare than "human" — because
falsely accusing a human writer is the more harmful error.

See [planning.md](planning.md) for the full design spec and architecture diagrams.

## Running it

```bash
pip install -r requirements.txt
echo "GROQ_API_KEY=your_key_here" > .env
python app.py            # serves on http://localhost:5000
```

### Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/submit` | Classify text. Body: `{ "text": "...", "creator_id": "..." }` |
| `POST` | `/appeal` | Contest a decision. Body: `{ "content_id": "...", "creator_reasoning": "..." }` |
| `GET`  | `/log` | View the structured audit log (JSON) |
| `GET`  | `/analytics` | Dashboard metrics as JSON |
| `GET`  | `/dashboard` | Human-facing analytics view (HTML) |

**Sample `/submit` response:**

```json
{
  "content_id": "3fab1cee-b46d-460e-b9e6-87a430920ebc",
  "attribution": "ai",
  "confidence": 0.72,
  "label": "We are confident this text was generated by AI.",
  "p_ai": 0.86,
  "llm_score": 0.9,
  "llm_reasoning": "Even rhythm and stock connectives throughout.",
  "stylo_score": 0.8,
  "stylo_metrics": { "ai_marker_count": 4, "sentence_length_cv": 0.21, "commas_per_sentence": 1.8 }
}
```

Both individual signal scores (`llm_score`, `stylo_score`) are returned next to
the fused `p_ai` so the decision is fully auditable, not a black box.

## Multi-signal detection pipeline

Provenance Guard scores every submission with **two independent signals** and
fuses them. Each measures a different property of the text and fails in a
different way, so combining them is more robust than either alone.

### Signal 1 — LLM-based classification (Groq)

A Groq-hosted `llama-3.3-70b-versatile` model is prompted as a forensic
text-attribution analyst and returns `P(AI) ∈ [0, 1]`.

- **What it captures:** holistic, semantic-level cues — generic phrasing, even
  rhythm, lack of an idiosyncratic voice, over-balanced structure, and hedging.
  It is the stronger signal because it reads the text the way a person would.
- **What it misses:** it works on word association and sequence predictability,
  so genuinely human writing that is *sarcastic, ironic, or subtly humorous* can
  read as "incoherent" and get flagged AI. Off-the-shelf LLM detectors are also
  biased toward calling **long, verbose** text AI — so the system prompt
  explicitly instructs the model to ignore length, verbosity, and vocabulary
  level and judge on style alone.

### Signal 2 — Stylometric heuristics (pure Python)

A dependency-free function that computes three measurable text statistics, maps
each to an "AI-ness" sub-score, and takes a weighted combination:

| Metric | What it measures | Direction |
|---|---|---|
| **AI-phrase markers** (weight 0.6) | density of LLM-favored connectives/clichés ("furthermore", "it is important to note", "leverage", "stakeholders") | more markers → leans AI |
| **Burstiness** (weight 0.25) | coefficient of variation of sentence lengths; humans vary sentence length, AI is more uniform | lower variance → leans AI |
| **Comma density** (weight 0.15) | commas per sentence; AI favors balanced, comma-rich constructions | higher density → leans AI |

- **What it captures:** surface-level structural fingerprints that don't require
  a model call — cheap, fast, and a useful sanity check on Signal 1.
- **What it misses:** it has no understanding of *meaning*. It can be fooled by
  formal-but-human prose (which is why phrase-markers, not burstiness, carry the
  most weight — burstiness alone wrongly ranks uniform academic writing as AI),
  and it needs enough text to be statistically meaningful (under ~10 words it
  falls back to a neutral `0.5`).

## Confidence scoring with uncertainty

### How the two signals combine

Each signal independently produces `P(AI) ∈ [0, 1]`. They are fused with a
weighted average that trusts the LLM more, since it is the stronger holistic
judge and stylometry is only a cheap sanity check:

```
final P(AI) = 0.6 · llm_score + 0.4 · stylo_score
```

We then separate **direction** from **confidence**, because a 0.51 call should
not look as sure as a 0.95 call:

- **Direction** — which side of 0.5 the score is on (`> 0.5` leans AI).
- **Confidence** — distance from the 0.5 decision boundary, rescaled to `[0, 1]`:
  `confidence = 2 · |P(AI) − 0.5|`. So `P(AI)=0.50` → confidence `0.00`
  (maximum ambiguity), `0.75` → `0.50`, and `0.95`/`0.05` → `0.90` (very sure).

Direction is mapped to a label using **asymmetric thresholds** — "AI" is harder
to declare than "human", giving human writers a wider buffer against false
accusation:

| Fused `P(AI)` | Attribution |
|---|---|
| `≤ 0.35` | **human** |
| `0.35 – 0.75` | **uncertain** |
| `≥ 0.75` | **AI** |

### How I validated that scores are meaningful

I tested inputs that *should* land in different bands and confirmed they do:

- **High-confidence AI:** verbose, marker-heavy text scores high on both signals
  (e.g. `llm 0.9 + stylo 0.8 → P(AI) 0.86`, confidence 0.72).
- **High-confidence human:** terse, idiosyncratic prose scores low on both
  (e.g. `llm 0.2 + stylo 0.09 → P(AI) 0.14`, confidence 0.69).
- **Low-confidence / uncertain:** cases where the two signals *disagree* — e.g.
  the LLM leans AI while stylometry leans human — pull the fused score toward
  0.5 and collapse confidence (e.g. `llm 0.7 + stylo 0.11 → P(AI) 0.46`,
  confidence 0.07). This is exactly the behavior we want: when the signals
  conflict, the system says so instead of guessing.

The thresholds (0.35 / 0.75) and the confidence tier (0.70, below) are starting
values; the asymmetry is a deliberate policy choice, not an arbitrary cutoff.

## Transparency label

The reader never sees a raw number — they see plain English. The label varies on
**both** the direction of the call **and** how confident the system is, so a
borderline result reads differently from a sure one (confidence `≥ 0.70` reads
"confident"; below that, "likely"):

| Case | Condition | Label text |
|---|---|---|
| **High-confidence AI** | `attribution = ai`, confidence ≥ 0.70 | "We are confident this text was generated by AI." |
| **Low-confidence AI** | `attribution = ai`, confidence < 0.70 | "This text was likely generated by AI, but we are not certain." |
| **High-confidence human** | `attribution = human`, confidence ≥ 0.70 | "We are confident this text was written by a human." |
| **Low-confidence human** | `attribution = human`, confidence < 0.70 | "This text was likely written by a human, but we are not certain." |
| **Uncertain** | `0.35 < P(AI) < 0.75` | "We can't confidently tell whether this text is AI-generated or human-written." |

No jargon — no "classifier output", no "logit", no "P(AI)" in the reader-facing
text. The wording changes with confidence, not just the underlying number.

## Appeals workflow

Creators can contest a classification via `POST /appeal` with the `content_id`
and their reasoning. The system:

1. Verifies the `content_id` refers to a real prior decision (404 otherwise).
2. Flips that content record's status to **`under review`**.
3. Appends an `appeal` event to the audit log, carrying the creator's reasoning
   **and a copy of the original decision's scores**, so the contest and the
   original sit side by side.

No automatic re-classification happens — an appeal hands the case to a human
reviewer. This is the system's answer to the *false-positive problem*: a human
writer wrongly flagged as AI has a concrete, logged path to recourse.

## Rate limiting

Every `/submit` triggers a Groq LLM call, which costs money and counts against
Groq's per-minute ceiling. The limits below protect cost, keep the app under
Groq's RPM limit, and block abuse — without getting in the way of real use on a
writing platform, where people analyze pieces occasionally, not in bursts.

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | 10/min, 100/day per IP | 10/min allows interactive testing but blocks scrapers; 100/day caps daily LLM cost and stays under Groq's free-tier RPM. |
| `POST /appeal` | 10/min per IP | Appeals are rare, deliberate human actions; a low limit discourages spam. |

When a limit is exceeded, Flask-Limiter returns **HTTP 429 Too Many Requests**
with a JSON error body, and the request never reaches the Groq call.

## Audit log

Every attribution decision — including the confidence score, both individual signal
scores, and any appeals — is captured in a structured audit log. The log is a SQLite
table (`audit_log` in `audit_log.db`) and is exposed as JSON at `GET /log`; it is the
single source of truth that both `/submit` and `/appeal` write to.

**Schema** — one row per *event*. A `/submit` writes a `classification` row; a `/appeal`
flips that content's status to `under review` and appends an `appeal` row that copies the
original decision's scores (so the decision and the contest sit side by side) and stores
the creator's reasoning.

| Field | Type | Meaning |
|---|---|---|
| `timestamp` | TEXT | ISO-8601 UTC time the event was recorded |
| `content_id` | TEXT | UUID linking a classification to its appeal(s) |
| `creator_id` | TEXT | Who submitted the content |
| `event` | TEXT | `classification` or `appeal` |
| `attribution` | TEXT | `human` / `ai` / `uncertain` |
| `confidence` | REAL | `2·|P(AI) − 0.5|`, in [0, 1] |
| `llm_score` | REAL | Signal 1 — Groq LLM P(AI) |
| `stylo_score` | REAL | Signal 2 — stylometric P(AI) |
| `status` | TEXT | `classified` → `under review` (after an appeal) |
| `creator_reasoning` | TEXT | The creator's argument (appeal rows only; `null` otherwise) |

> Whether an appeal has been filed is captured two ways: the original classification row's
> `status` flips to `under review`, and a dedicated `event = "appeal"` row is appended.

**Sample — live `GET /log` output** (4 entries: three classifications spanning all label
variants, plus the appeal filed against the AI-flagged one):

## Analytics dashboard

`GET /dashboard` renders a browser view (and `GET /analytics` returns the same
numbers as JSON) computed from the audit log — the single source of truth both
endpoints already write to. It surfaces **three metrics**:

| Metric | What it shows |
|---|---|
| **Detection pattern** | Verdict counts (ai / human / uncertain) and the AI-vs-human ratio — the ratio of AI to human verdicts. |
| **Appeal rate** | Appeals filed per classification. |
| **Mean confidence by verdict** | Average confidence within each verdict, so a confident "ai" call is distinguishable from a borderline one. |

Because the dashboard reads the same SQLite table `/submit` and `/appeal` write
to, the numbers can never drift from the log.

## Known limitations

The signals only measure **semantics** (Signal 1) and **surface structure**
(Signal 2), so any text whose human-ness lives outside those two channels is at
risk of misclassification:

- **Sarcastic, ironic, or subtly humorous human writing** is the clearest
  failure case. The LLM signal judges "coherence", and deadpan or absurdist
  human prose deliberately violates coherence — so it can read as AI even though
  a person wrote it.
- **Highly formal human writing** (academic abstracts, legal prose, corporate
  style-guide copy) trips the stylometric signal: it is uniform (low burstiness)
  and connective-heavy (many phrase markers), exactly the fingerprint the
  heuristics associate with AI. Phrase-markers are weighted highest specifically
  to reduce this, but it is not eliminated.
- **Very short submissions** (under ~10 words) give the stylometric signal too
  little data, so it returns a neutral `0.5` and the decision leans entirely on
  the LLM.

These are inherent to *what the signals can see*, not bugs — which is exactly
why the appeals workflow exists.

## Spec reflection

The biggest divergence from [planning.md](planning.md) is the **transparency
label**. The plan specified three labels keyed purely on direction — one each
for AI, human, and uncertain — with the AI label reading *"We are confident this
text is AI."* During implementation I realized that wording was dishonest at the
edge: a submission at `P(AI) = 0.75` clears the "AI" threshold but only has a
confidence of `0.50`, so telling the reader we are "confident" overstates what
the system actually knows. I split each directional label into a **confidence
tier** (`confident` vs `likely`, divided at confidence `0.70`), so a borderline
AI call now reads *"This text was likely generated by AI, but we are not
certain."* This also better satisfies the project's own goal that the label
*text* — not just a number — should reflect certainty. The thresholds and fusion
weights otherwise matched the plan.

## AI usage
**Building the stylometric signal and fixing the label-confidence mismatch.**
I asked the AI to implement the first and second signal from the spec. Its initial draft used type-token ratio (vocabulary diversity) as a metric; **I decided against
it** after seeing that modern AI has lexical diversity as high as humans, so it didn't discriminate — I replaced it with comma density and re-weighted the phrase-marker metric highest. Separately, when reviewing the transparency label,
the AI flagged that the label said "confident" even for borderline AI calls; **I directed it to add a confidence tier** so the wording (`confident` vs
`likely`) tracks the actual confidence score rather than just the direction (see the spec reflection above).
